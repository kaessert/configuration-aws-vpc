## What Was Accomplished

### Problem Solved
IPv6 dual-stack and IPv6-native VPC creation was failing with `ParseIntError` during E2E tests. The composition function attempted to set `assignIpv6AddressOnCreation=true` on subnets before AWS assigned IPv6 CIDR blocks, causing AWS API errors.

### Solution Implemented
Implemented late-binding pattern for IPv6 subnet CIDR calculation:

1. **First composition pass:** VPC created with `assignGeneratedIpv6CidrBlock=true`, subnets created without IPv6 CIDRs
2. **Second composition pass:** Composition extracts AWS-assigned IPv6 CIDR from VPC status, calculates subnet /64 blocks, updates subnets with IPv6 configuration

## Technical Details

### Late-Binding Mechanism

**VPC IPv6 Status Extraction:**
```kcl
_vpcResources = [ocds[name].Resource for name in ocds if ocds[name]?.Resource?.kind == "VPC"] if ocds else []
_hasVpc = len(_vpcResources) > 0
_hasStatus = _hasVpc and "status" in _vpcResources[0] and _vpcResources[0]["status"]
_hasAtProvider = _hasStatus and "atProvider" in _vpcResources[0]["status"]
_hasIpv6Cidr = _hasAtProvider and "ipv6CidrBlock" in _vpcResources[0]["status"]["atProvider"]
observedVpcIpv6CidrBlock = _vpcResources[0]["status"]["atProvider"]["ipv6CidrBlock"] if _hasIpv6Cidr else ""
```

**IPv6 CIDR Calculation:**
- AWS VPC IPv6: /56 CIDR block (e.g., `2600:1f13:d6f:5800::/56`)
- AWS Subnet IPv6: /64 CIDR blocks (e.g., `2600:1f13:d6f:5800::/64`, `2600:1f13:d6f:5801::/64`)
- Function calculates by manipulating segment 3's lower 8 bits

### Bug Fixed

**Original Issue:**
```
ParseIntError { kind: InvalidDigit }
at /src/subnets.k:78:1
```

**Cause:** IPv6 address parsing used Python-style `while` loops and list `.append()` which aren't supported in KCL.

**Fix:** Rewrote using KCL-compatible functional programming with list comprehensions and concatenation.
